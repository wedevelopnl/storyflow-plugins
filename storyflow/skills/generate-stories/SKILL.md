# Story Generation Guidelines

> **Source of truth**: `ai-service/skills/story-generation/story-generation.md` in the StoryFlow repository.
> This skill is adapted for the Claude Code plugin where stories are generated by the Software Architect locally and submitted via MCP tool.

## Audience

Stories are read by non-technical clients who need to understand what they are getting and what it costs. They do NOT need to know how it will be built. The development team handles technical details during refinement.

## Language Guardrails

FORBIDDEN in story titles and descriptions:
- Framework or library names (React, Symfony, PostgreSQL, Redis, etc.)
- Technical protocols (REST, GraphQL, WebSocket, SSE, JWT, OAuth, etc.)
- Code concepts (API endpoint, migration, middleware, controller, service, entity, etc.)
- Architecture patterns (microservice, event-driven, CQRS, etc.)
- Developer tools (Docker, Git, webpack, npm, etc.)

USE INSTEAD:
- "Real-time updates" NOT "WebSocket connection"
- "Secure login" NOT "JWT authentication"
- "Automatic notifications" NOT "Event-driven email triggers"
- "Data import" NOT "CSV parsing endpoint"
- "Search functionality" NOT "Elasticsearch integration"

SELF-CORRECTION CHECKPOINT:
Before finalizing each story, ask yourself:
1. "Would a non-technical client understand this title?"
2. "Am I describing WHAT the user gets, or HOW it will be built?"
3. "Could this title appear on an invoice to the client?"

If the answer to #1 or #3 is NO, or #2 is "HOW": REWRITE in business terms.

## Story Grouping (CRITICAL)

GROUP by user journeys or business workflows, NOT by technical components.

### Wrong (AVOID)
- "User entity validation" / "Customer entity validation" / "Order entity validation" (per-entity splitting)
- "Set up audit logging infrastructure" (technical framing)
- "API endpoint for user management" (implementation detail)

### Correct
- "Activity history: see who changed what and when" (user-facing feature)
- "Form validation with clear error messages" (1 story covering all forms)
- "User management: invite, edit roles, and deactivate users" (complete workflow)

### Rules
- If the same functionality applies to multiple areas: 1 story (mention all areas in the description)
- Split by difference in USER EXPERIENCE, not difference in technical approach
- Group stories that together form a complete user journey into one epic
- Mention in the description which areas or objects are affected
- Every story MUST describe end-user value. Stories about infrastructure, tooling, or invisible backend changes are not permitted.

## Story Description Format

Each story has a markdown description with the following structure:

```markdown
**As a** [user role]
**I want to** [action/feature]
**So that** [business value]

**Business context:**
[Relevant business background, related workflows, constraints the client should know about]

**Acceptance Criteria:**

**Scenario: [name]**
- **Given** [precondition]
- **When** [action]
- **Then** [expected result]
- **And** [additional expected result]

**Scenario: [name]**
- **Given** [precondition]
- **When** [action]
- **Then** [expected result]
```

### Complete Example

**Title:** "Email notifications for order status changes"

**Description:**
```
**As a** registered customer
**I want to** receive email notifications when my order status changes
**So that** I stay informed about my delivery without having to check the app

**Business context:**
Customers currently have to log in to see if their order moved to a new status.
Automatic notifications reduce support inquiries and improve the customer experience.

**Acceptance Criteria:**

**Scenario: Order shipped notification**
- **Given** I have placed an order and it is confirmed
- **When** the order status changes to "Shipped"
- **Then** I receive an email with the tracking number and estimated delivery date

**Scenario: Notification preferences**
- **Given** I am on my account settings page
- **When** I disable order notifications
- **Then** I no longer receive status change emails for future orders

**Scenario: Invalid email address**
- **Given** my account email address is invalid or bouncing
- **When** the system tries to send a notification
- **Then** the notification is flagged for review and I see a prompt to update my email next time I log in
```

**Complexity:** M
**Priority:** medium

## Story Requirements

### User Story Format
- **As a**: Use the EXACT persona names from the briefing's User Personas section (e.g., "warehouse manager", "shift supervisor", "online shopper"). Fall back to generic roles (end user, admin, manager) ONLY if the briefing does not define specific personas. NEVER use technical actors ("database", "API", "system").
- **I want to**: What action or feature? (concrete, specific, in the user's language)
- **So that**: What PROBLEM does this solve, or what OUTCOME does this enable? (business value, NOT technical details)

### Good story title examples
- "Activity history: see who changed what and when"
- "Email notifications for order status changes"
- "Dashboard with key business metrics"
- "Bulk import of customer data from spreadsheet"
- "Two-factor authentication for secure login"

### Bad story title examples (AVOID)
- "Set up audit logging infrastructure" (too technical)
- "Implement WebSocket event handlers" (implementation detail)
- "Database migration for user preferences" (invisible to the client)
- "Refactor authentication middleware" (internal concern)

### Strong "So that" examples
- "So that I can identify overdue invoices before they become collection cases" (risk mitigation)
- "So that new employees can start working independently within their first day" (efficiency gain)
- "So that I do not need to manually reconcile data between two systems every week" (time saving)

### Weak "So that" examples (IMPROVE THESE)
- "So that I can manage users" (restates the want, no value)
- "So that the data is up to date" (too vague, no measurable impact)
- "So that I have an overview" (does not explain why the overview matters)

## Acceptance Criteria (Given-When-Then)

Each story has 2-5 scenarios in the description:
- At least 1 happy path scenario
- At least 1 error/edge case scenario
- Be specific: "within 2 seconds" instead of "fast"
- No OR statements (split into separate scenarios)
- Write scenarios from the user's perspective, not from a technical perspective

### Writing testable scenarios
- **Given**: Include specific data states, not just "I am logged in." Example: "Given I have 3 pending orders and 1 completed order"
- **When**: Describe observable user actions, not internal processes. Example: "When I select 'Last 30 days' from the date filter"
- **Then**: State observable outcomes that can be verified visually or by assertion. Example: "Then I see only the 3 pending orders, sorted by date descending"
- Every scenario must be verifiable without looking at the code or database

### Error scenario specificity
Each error scenario MUST specify:
- The exact error condition (not just "something goes wrong")
- The expected feedback shown to the user
- What data is preserved or lost when the error occurs

### Common edge cases to consider for each story
- What happens with no data (empty state)?
- What happens with large amounts of data?
- What if the user does not have permission?
- What if required input is missing or invalid?

## Complexity

Rate the scope of each story. This is about how much new functionality the client gets, not about developer hours.

- **S** = Single-screen change, one user interaction. Examples: "Add a search filter to the order list", "Show a confirmation message after saving"
- **M** = New screen or workflow with 2-3 user interactions. Examples: "Contact form with validation and confirmation", "Export reports as PDF"
- **L** = Multi-screen workflow or feature with 4+ interactions, multiple user roles, or complex business rules. Examples: "User onboarding wizard (registration, verification, profile setup)", "Role-based dashboard with different views per role"
- **XL** = Too large. If a story spans multiple independent user journeys, SPLIT it. Each resulting story should be independently demonstrable to the client.

When in doubt between two sizes, choose the larger one.

## Priority

Assess each story based on its importance for the client's goals:

- **high** = Must-have: The product cannot launch or fulfill its core purpose without this. Blocking for other stories or critical for the primary user journey.
- **medium** = Should-have: Important for a complete product experience, but the product is functional without it. Can be delivered in a fast follow-up.
- **low** = Nice-to-have: Adds polish or convenience. The client would not delay launch for this.

Distribution guideline: If more than 40% of stories are high priority, reconsider whether they are all truly launch-critical. Effective prioritization means making hard choices.

Base priority on:
1. How many users are affected (all users vs. admin-only)
2. What happens if this is deferred (product broken vs. minor inconvenience)
3. Whether the briefing explicitly emphasized this as critical

## Epics

Group related stories into epics based on user journeys or business workflows:
- Only create an epic if it will contain 3 or more related stories
- Target 3-7 stories per epic. More than 7 suggests the epic covers multiple distinct journeys and should be split.
- Give each epic a clear, business-oriented title and a brief description explaining what user journey this epic completes
- Stories that do not belong to an epic go into standaloneStories

## Story Count Guidance

- A short briefing (1-2 pages) typically yields 3-8 stories
- A medium briefing (3-5 pages) typically yields 8-20 stories
- A long briefing (5+ pages) typically yields 15-35 stories
- If you exceed 40 stories, consider whether you are splitting too granularly
- If you produce fewer than 3 stories, consider whether you are grouping too aggressively

## Output Format

Structure your stories as JSON matching the `create-briefing-stories` MCP tool input:

```json
{
  "epics": [
    {
      "title": "Epic title",
      "description": "What user journey this epic completes",
      "stories": [
        {
          "title": "Story title",
          "description": "Full markdown description (As a / I want / So that + Business context + Acceptance Criteria)",
          "complexity": "M",
          "priority": "medium"
        }
      ]
    }
  ],
  "standaloneStories": [
    {
      "title": "Story title",
      "description": "Full markdown description",
      "complexity": "S",
      "priority": "low"
    }
  ]
}
```

## Order

1. Core features that deliver the most value first
2. Then enhancements and extensions
3. Nice-to-haves last
